# CLAUDE.md — FridgeMind: PWA для планирования меню и продуктов

## 1. Идея продукта

PWA-приложение для планирования питания и продуктов дома. Общее на household (семья/несколько человек, без ограничения по числу участников) — запасы, рецепты, меню и список покупок делятся между всеми:

- ведут список продуктов, которые есть дома (вручную или через фото холодильника, которое распознаёт ИИ);
- хранят рецепты с ингредиентами;
- собирают меню на неделю из рецептов;
- по каждому дню видят, каких ингредиентов не хватает (сравнение "есть дома" vs "нужно по рецепту");
- формируют список покупок на основе недостающих ингредиентов — список общий и обновляется в реальном времени, так что один может дописать позицию, пока второй уже в магазине.

Ключевая фишка — фото холодильника → ИИ распознаёт продукты → автоматически обновляет "домашний инвентарь".

## 2. Название

**FridgeMind** — выбрано.

Перед запуском стоит проверить занятость домена (fridgemind.app / .com) и никнейма в соцсетях/сторах, но для personal-use MVP это не блокер.

## 3. Функциональный объём

### MVP (первая версия)
1. Регистрация/вход (email или соц. вход) + **household**: приложение общее на семью — один регистрируется первым и создаёт household, остальные присоединяются по приглашению (ссылка/код), сколько бы их ни было. Все данные (запасы, меню, рецепты, список покупок) общие для household, а не привязаны к отдельному пользователю. Внутри household — **3 роли**: Организатор (управляет составом семьи), Редактор (рецепты, меню, слоты приёма пищи), Участник (всё остальное — инвентарь, список покупок, отметка "скушано" — без ограничений). Экран **Профиль/Настройки** — список участников с ролями, кнопка "пригласить", управление слотами приёма пищи (drag-and-drop), выход из household/аккаунта (подробности в разделе 6).
2. CRUD рецептов: название, фото, базовое число порций, примерное время приготовления, ингредиенты (выбор из справочника продуктов, количество, единица), **шаги приготовления по одному, у каждого шага — текст и опционально своя фотография**, бейджи способа приготовления (сковорода/духовка/мультиварка/гриль/микроволновка/без готовки). Число порций можно менять при просмотре рецепта и при добавлении в конкретный день меню — количество ингредиентов пересчитывается пропорционально. **Редактирование и удаление** рецепта (удаление — с подтверждением, т.к. необратимо; если рецепт используется в будущих неделях меню — предупреждение об этом перед удалением). **Поиск** по названию рецепта и по названию ингредиента (простое текстовое поле над списком) — отдельно от фильтра "что приготовить из того, что есть" (тот сортирует по совпадению с инвентарём, а не ищет по тексту).
3. Домашний инвентарь продуктов: добавление вручную (выбор продукта из справочника или создание нового, количество; категория берётся из справочника), **редактирование количества/единицы уже добавленного продукта и удаление позиции целиком** (например, если продукт закончился раньше, чем ожидалось, или добавлен по ошибке). **Поиск** по названию продукта — актуально, когда список инвентаря разрастается и пролистывать все категории неудобно.
4. Фото холодильника — **можно сделать/выбрать несколько фото за один раз** (например, отдельно холодильник, отдельно морозилка и полка с крупами), не только одно → все фото уходят в Claude Vision одним запросом → распознанный и объединённый (без дублей) список продуктов → пользователь подтверждает/правит → сохраняется в инвентарь.
5. Конструктор меню на неделю: выбор рецептов по дням с полностью настраиваемыми слотами приёма пищи. Household сам называет, добавляет, удаляет и сортирует свои слоты (по умолчанию — "Завтрак/Обед/Ужин", дальше можно переименовать или добавить, например, "Перекус") в Профиле, порядок задаётся drag-and-drop'ом и виден всем членам household одинаково. Назначенный на слот рецепт можно **убрать** (не только заменить другим) — слот возвращается в пустое состояние "+ добавить рецепт".
6. Просмотр дня: список нужных ингредиентов (по всем слотам дня, куда назначен рецепт) + отметка "есть дома" / "нужно купить".
7. Автогенерация списка покупок на неделю (агрегация недостающих ингредиентов по всем дням, с суммированием количеств) + возможность дописать туда ручные позиции с выбором категории из фиксированного списка (молочка/овощи/бытовая химия и т.д., см. раздел 5) — список группируется по категориям для удобства в магазине. **Фильтр по дням**: можно смотреть список не только на всю неделю, а на сегодня, на завтра, или на произвольно выбранные несколько дней — количества пересчитываются под выбранные дни (подробности в разделе 6).
8. "Что приготовить из того, что есть" — список рецептов, отсортированный по проценту совпадения ингредиентов с текущим инвентарём (сколько ингредиентов рецепта уже есть дома из общего числа). Ключевая фича, связывающая инвентарь, распознавание фото и рецепты в одну ценность.
9. Отметка "скушано" на приёме пищи применяется сразу и без подтверждений; отдельно, необязательно — предложение списать ингредиенты из инвентаря (только количество по рецепту/порциям, без сравнения с остатками), с степпером числа порций прямо на этом экране (вдруг готовили не столько, сколько планировали — количества пересчитаются) и возможностью правки итоговых цифр перед подтверждением или полным пропуском этого шага.
10. Отметка "куплено" в списке покупок — мгновенная, без всплывающего окна на каждую позицию (удобно, когда отмечаешь много позиций подряд прямо в магазине); позиции можно редактировать (название, количество, единица) прямо в списке. Отдельно — кнопка **"Добавить в запасы"**, которая разом собирает все отмеченные "куплено" (и ещё не перенесённые в инвентарь) позиции, показывает их одним списком с возможностью поправить количество, и по подтверждению добавляет всё в `PantryItem` за один раз (подробности в разделе 6).
11. PWA: устанавливается на телефон, работает офлайн (просмотр сохранённого меню/рецептов/списка покупок), иконка на экране.
12. **Список покупок в реальном времени**: все члены household видят изменения друг друга без перезагрузки — кто-то добавил позицию из дома, другой уже в магазине увидел её на экране; отметки "куплено" синхронизируются мгновенно у всех.
13. **Поделиться списком покупок текстом** — кнопка "Поделиться" на экране "Список покупок" (рядом с фильтром по дням), формирует список в виде обычного текста, сгруппированного по категориям (как он выглядит в самом приложении), и открывает нативный Web Share API (тот же механизм, что и для инвайт-ссылки в разделе 6) — чтобы отправить список тому, у кого нет приложения (например, отправить бабушке в мессенджер). Учитывает текущий фильтр по дням — делится именно тем, что видно на экране в момент нажатия, а не всегда всей неделей.
14. **Тёмная тема** — переключатель Светлая/Тёмная/Как в системе в экране "Профиль/Настройки". Это личная настройка устройства, а не household — хранится локально (например, `localStorage`), не синхронизируется между участниками семьи и не относится к общим данным household.

## 4. Технологический стек

| Слой | Выбор | Почему |
|---|---|---|
| Frontend/фреймворк | **Next.js (App Router, TypeScript)** | Один стек для UI, API-роутов и SSR; отличная поддержка PWA и деплоя на Vercel |
| PWA-слой | **Serwist (`@serwist/next`)** | Актуальный преемник next-pwa в 2026 году, из коробки даёт service worker, precaching, офлайн-стратегии без ручного написания SW |
| UI | **Tailwind CSS + shadcn/ui** | Быстрая сборка адаптивного интерфейса, хорошо ложится на мобильный UX |
| База данных | **PostgreSQL через Supabase** | Даёт БД + Auth + Storage (для фото холодильника/рецептов) + реалтайм в одном сервисе, щедрый бесплатный тариф |
| ORM | **Prisma** | Зрелая миграционная система, автогенерация типов из схемы, `@map`/`@@map` для маппинга camelCase ↔ snake_case из коробки; Claude Code генерирует по нему код надёжнее, чем по менее распространённым альтернативам |
| Локальное хранилище/офлайн | **IndexedDB через Dexie.js** | Кэш меню, рецептов и списка покупок для офлайн-режима, синхронизация при восстановлении сети |
| Drag-and-drop | **@dnd-kit/core** | Для сортировки слотов приёма пищи (и при желании — шагов рецепта) на телефоне; современная, доступная (accessible), активно поддерживаемая библиотека, работает и с touch-событиями |
| ИИ-распознавание фото | **Claude API (vision)** | Не нужен отдельный специализированный сервис распознавания еды — Claude принимает фото и возвращает структурированный список продуктов (JSON) через один и тот же API, что упрощает стек |
| Генерация меню/рецептов (опционально) | **Claude API (текст)** | Подсказки рецептов на основе того, что есть дома |
| Аутентификация | **Supabase Auth** | Встроена в тот же сервис, что и БД/Storage |
| Real-time синхронизация | **Supabase Realtime** | Подписка на изменения таблиц Postgres по вебсокету "из коробки" — нужно для общего списка покупок между всеми членами household, без отдельной инфраструктуры |
| Хостинг | **Vercel** | Нативная поддержка Next.js, простой деплой PWA |

Данные из поиска подтверждают: в 2026 году Serwist — стандартный выбор для PWA на Next.js (next-pwa считается устаревшим), а Claude Vision API подходит для задач распознавания еды на фото как универсальное решение без отдельного computer-vision сервиса.

**Prisma vs Drizzle**: изначально предлагался Drizzle (легче, без Rust-движка, чуть быстрее холодный старт на edge-рантайме). Заменили на **Prisma** по запросу — для этого проекта (обычные serverless-функции на Vercel, не edge runtime) разница в холодном старте некритична, а Prisma даёт более зрелые миграции и генерацию типов, и Claude Code стабильнее пишет код под неё, так как это более распространённый инструмент.

## 5. Модель данных (черновик)

- **Household**: id, name (опционально, например "Наша кухня"), inviteCode (для приглашения новых участников, без ограничения по количеству)
- **User**: id, email, name, householdId (→ Household), role (→ `HouseholdRole`) — пользователь всегда принадлежит ровно одному household, роль определяет, что ему доступно (см. "Роли в household" ниже)
- **Ingredient** (справочник): id, name, defaultUnitType, category (→ `ProductCategory`) — общий каталог продуктов; и рецепты, и инвентарь ссылаются на него, а не хранят название свободным текстом. Справочник глобальный (не привязан к household), чтобы не дублировать общеупотребимые продукты
- **Recipe**: id, householdId, title, photoUrl, baseServings (порции, на которые рассчитаны количества в `RecipeIngredient`), cookTimeMinutes (примерное время приготовления в минутах, необязательное), cookingMethods (массив значений `CookingMethod` — теги для бейджей, см. ниже), deletedAt (nullable — soft-delete, чтобы удаление рецепта не стирало историю "скушано" в прошлых неделях; см. "Удаление рецепта" ниже)
- **RecipeStep**: id, recipeId, order, instruction, photoUrl (nullable) — рецепт хранится как упорядоченный список шагов, у каждого свой текст и опциональное фото
- **RecipeIngredient**: id, recipeId, ingredientId (→ Ingredient), quantity, unit
- **PantryItem** (домашний инвентарь): id, householdId, ingredientId (→ Ingredient), quantity, unit, addedVia (manual/photo)
- **MenuWeek**: id, householdId, weekStartDate
- **MenuDay**: id, menuWeekId, date
- **MealSlot**: id, householdId, name (произвольный текст, например "Завтрак" или "Перекус на работе"), order (порядковый номер, задаётся drag-and-drop'ом) — household сам определяет набор и порядок приёмов пищи, а не выбирает из фиксированного списка
- **MenuDayMeal**: id, menuDayId, mealSlotId (→ MealSlot), recipeId, servings (сколько порций готовим в этот раз — по умолчанию равно `Recipe.baseServings`, но можно поменять при планировании), isEaten, eatenAt, eatenByUserId — по одной записи на каждый слот дня, в который назначен рецепт
- **ShoppingListItem**: id, householdId, weekId, ingredientId (nullable — см. ниже), name, quantity (суммарное количество на всю неделю), unit, isBought, addedToPantry (уже перенесено в инвентарь или нет — см. "Массовое обновление инвентаря" в разделе 6), isManual, manualCategory (→ `ProductCategory`, nullable — заполняется только для ручных позиций, см. ниже), boughtByUserId (nullable — кто из household отметил купленным)
- **ShoppingListItemMeal**: id, shoppingListItemId (→ ShoppingListItem), menuDayMealId (→ MenuDayMeal), quantity (вклад именно этого приёма пищи в общее количество) — связывает позицию списка с конкретными днём/приёмом пищи, чтобы можно было отфильтровать список по дням (см. "Фильтр по дням" в разделе 6). У ручных позиций (`isManual`) таких связей нет — они не привязаны к дням.

Всё, что раньше было привязано к `userId` (Recipe, PantryItem, MenuWeek, ShoppingListItem, MealSlot), теперь принадлежит `householdId` — весь функционал общий для всех членов household. Индивидуальным остаётся только сам `User` (логин/имя) и точечные поля вроде `eatenByUserId`/`boughtByUserId`, которые просто фиксируют, кто именно совершил действие — для истории, не для разграничения доступа.

### Household и приглашение участников

- Первый пользователь при регистрации автоматически создаёт новый `Household`.
- В настройках есть экран "пригласить" — генерируется `inviteCode`/ссылка; любой, кто перейдёт по ней, регистрируется (или входит, если уже есть аккаунт) и присоединяется к тому же `householdId` вместо создания своего. Число участников не ограничено — архитектура (общий `householdId` на всех записях + RLS) одинаково работает для 2 человек и для большой семьи.
- Доступ к данным на бэкенде проверяется через `householdId` пользователя (Supabase RLS-политики: строка видна/редактируема, если её `householdId` совпадает с household текущего пользователя), а для отдельных таблиц — ещё и через роль (см. ниже).

### Роли в household

Пересмотрено: роли нужны уже в MVP, не все участники равноправны. Три роли (`HouseholdRole`):

- **ORGANIZER** (Организатор) — всё, что может Редактор, плюс управление составом семьи: приглашать и удалять участников, назначать/менять роли другим. Первый пользователь, создавший household, автоматически становится Организатором. Организаторов в household может быть несколько (Организатор может назначить Организатором кого-то ещё).
- **EDITOR** (Редактор) — всё, что может Участник, плюс: создание/редактирование/удаление рецептов, построение и изменение меню на неделю (назначать/убирать рецепты по дням, менять число порций), управление слотами приёма пищи (`MealSlot`: добавить/переименовать/удалить/порядок).
- **MEMBER** (Участник) — просмотр всего в household (рецепты, меню, инвентарь, список покупок); работа с домашним инвентарём (добавление вручную и через фото, редактирование, удаление); отметка "скушано" и связанное с ней списание запасов; список покупок — **полностью без ограничений** (отметка "куплено", редактирование позиций, добавление ручных позиций, массовый перенос купленного в инвентарь). Не может: создавать/редактировать/удалять рецепты, менять меню на неделю, управлять слотами приёма пищи, приглашать/удалять участников или менять роли.

Раз Участник не может назначать рецепты на слоты, пустые слоты (без назначенного `MenuDayMeal`) для него просто **не отображаются** на экранах "Сегодня" и "Меню на неделю" — ему незачем видеть пустой слот "+ добавить рецепт", которым он всё равно не может воспользоваться. У Организатора и Редактора пустые слоты видны как обычно (с возможностью назначить рецепт).

Новый участник, присоединившийся по ссылке-приглашению, по умолчанию получает роль **MEMBER** — Организатор при необходимости повышает его до Редактора или до Организатора отдельно, уже после присоединения (осознанное решение, а не автоматическая раздача прав всем, кто просто перешёл по ссылке).

Ограничение: у household должен быть хотя бы один Организатор. Если Организатор — единственный в этой роли и хочет покинуть household (при наличии других участников), сначала нужно назначить Организатором кого-то ещё — иначе управлять составом семьи будет некому. Если Организатор — единственный участник household вообще, кнопка "Покинуть household" недоступна (как и раньше, см. ниже).

На уровне Supabase RLS: таблицы `Recipe`, `RecipeStep`, `RecipeIngredient`, `MenuDayMeal`, `MealSlot` разрешают запись только пользователям с ролью `ORGANIZER`/`EDITOR` в соответствующем household; управление `Household`/участниками (приглашение, удаление, смена ролей) — только `ORGANIZER`. Остальные таблицы (`PantryItem`, `ShoppingListItem`) разрешают запись всем ролям, как и раньше — проверяется только совпадение `householdId`, без дополнительной проверки роли.

### Слоты приёма пищи (MealSlot) — динамические, не фиксированный список

Изначально предполагался фиксированный enum (завтрак/второй завтрак/обед/полдник/ужин) с переключателями "включено/выключено". Пересмотрено: household сам создаёт, называет, удаляет и **сортирует drag-and-drop'ом** свои слоты приёма пищи — никакого захардкоженного списка типов.

- При создании household автоматически создаётся стартовый набор из трёх слотов — "Завтрак", "Обед", "Ужин" (order 0/1/2) — чтобы не начинать с пустого экрана. Дальше household может переименовать любой, добавить новый (например, "Перекус на работе"), удалить ненужный или перетащить в другом порядке.
- Порядок слотов (`order`) задаётся перетаскиванием в экране "Профиль/Настройки" и определяет порядок отображения на экранах "Сегодня" и "Меню на неделю" — редактируется один раз в настройках, а не отдельно на каждом экране.
- Удаление слота, к которому уже привязаны `MenuDayMeal` в существующих неделях — простое поведение для MVP: слот помечается как удалённый (soft-delete), старые записи остаются видны в прошлых неделях, но слот больше не предлагается при планировании новых.
- **Защита от дублей при пересоздании**: имя слота должно быть уникальным (без учёта регистра) среди **активных** (не удалённых) слотов household — нельзя создать или переименовать слот в имя, которое уже занято. Если пользователь добавляет новый слот с именем, совпадающим с именем ранее **удалённого** слота (например, удалил "Ужин", а потом завёл его заново) — вместо создания новой записи нужно **восстановить старую** (снять `deletedAt`, поставить в конец текущего порядка) и переиспользовать тот же `id`. Без этого правила два разных `MealSlot` с одинаковым именем "Ужин" (старый — ещё привязанный к текущей/будущей неделе, новый — пустой) окажутся видны одновременно на "Сегодня"/"Меню на неделю" в один и тот же день, что выглядит как дублирование. Проверка — на уровне server action/API при создании и переименовании слота, отдельного ограничения в БД не требуется.

### Удаление рецепта — soft-delete (симметрично MealSlot)

Пересмотрено (GitHub issue #5): жёсткое удаление рецепта стирало **все** `MenuDayMeal`, ссылающиеся на него, включая уже отмеченные `isEaten = true` в прошлых неделях (и, каскадом, историческую разбивку `ShoppingListItemMeal`). Это противоречило философии soft-delete, уже принятой для `MealSlot`. Приведено к тому же паттерну:

- У `Recipe` есть поле `deletedAt DateTime?`. "Удаление" — это `UPDATE recipes SET deleted_at = now()`, строка рецепта с её `RecipeStep`/`RecipeIngredient` остаётся в БД, чтобы eaten-записи прошлых недель сохранили полную историю (название, ингредиенты, шаги).
- **Гибрид**: будущие (не съеденные, `isEaten = false`) `MenuDayMeal` при удалении всё же убираются из плана (`deleteMany`) — рецепт исчезает из будущих слотов меню; eaten-записи не трогаются.
- Soft-deleted рецепт скрыт везде, где перечисляются/предлагаются активные рецепты (список "Рецепты", "что приготовить из того, что есть", `GET /api/recipes`, будущий пикер рецептов для меню) — через фильтр `deletedAt: null`. Редактирование soft-deleted рецепта заблокировано (scope `deletedAt: null` в `updateRecipe`).
- Экран просмотра `/recipes/[id]` **открывает** soft-deleted рецепт read-only (`getRecipeDetail(..., { includeDeleted: true })`) — чтобы ссылка из истории меню работала; кнопок редактирования/удаления на этом экране нет.
- RLS не меняется: soft-delete идёт через `UPDATE`, уже покрытый политикой `recipes_update_organizer_editor`; select-политика по household корректно отдаёт удалённый рецепт для read-only истории.

### Порции (servings)

У рецепта есть базовое число порций (`baseServings`), под которое рассчитаны количества в `RecipeIngredient` (например, "на 4 порции"). При добавлении рецепта в конкретный слот меню (`MenuDayMeal.servings`) можно поменять число порций для этого конкретного раза — например, обычно готовят на 4, а в этот день нужно на 2.

- Пересчёт простой пропорциональный: количество ингредиента для конкретного дня = `RecipeIngredient.quantity × (MenuDayMeal.servings / Recipe.baseServings)`. Округление — до целого для COUNT (штук нельзя купить дробно), без строгого округления для WEIGHT/VOLUME (примерная оценка и так не точная, см. раздел 7, пункт 2).
- Этот пересчитанный (а не базовый) объём участвует и в сравнении "есть/нужно" по дню, и в агрегации списка покупок, и в списании инвентаря при отметке "скушано".
- На экране рецепта (просмотр, не только при планировании недели) тоже показывается степпер количества порций — чтобы посмотреть "а сколько нужно ингредиентов, если готовить не на 4, а на 6" без привязки к конкретному дню меню, чисто для справки.
- "Что приготовить из того, что есть" (раздел 3, MVP-пункт 8) считает процент совпадения по наличию ингредиентов, не по количеству — смена порций на этот процент не влияет.

### Способ приготовления (бейджи)

У рецепта может быть один или несколько тегов способа приготовления — показываются бейджами на карточке рецепта в списке. Контролируемый список (`CookingMethod`), а не свободный текст — чтобы бейджи были визуально консистентны (у каждого свой значок):

- **STOVETOP** — плита/сковорода
- **OVEN** — духовка
- **MULTICOOKER** — мультиварка
- **GRILL** — гриль
- **MICROWAVE** — микроволновка
- **NO_COOK** — без готовки (салаты, бутерброды, сборка)

`Recipe.cookingMethods` — массив значений этого перечисления (рецепт может использовать и духовку, и плиту одновременно). Выбирается мультиселектом при создании/редактировании рецепта, не обязателен для заполнения.

Бейджи показываются не только на карточках экрана "Рецепты", но и на карточках приёмов пищи экрана "Сегодня" — рядом с `cookTimeMinutes` (примерным временем приготовления), чтобы сразу было видно и сколько времени, и что понадобится (плита/духовка/мультиварка и т.д.), не заходя в сам рецепт.

### Справочник ингредиентов (Ingredient catalog)

Сопоставление "есть дома / нужно купить" по свободному тексту ненадёжно ("молоко 3.2%" ≠ "молоко"), поэтому вводим единый справочник продуктов:

- Все продукты (в рецептах, в инвентаре, при распознавании фото) ссылаются на запись в таблице `Ingredient`, а не хранят название как произвольную строку.
- При добавлении нового продукта (вручную или через фото) — пользователь выбирает существующий пункт справочника через автокомплит/поиск, либо создаёт новый (справочник растёт органически, начиная пустым или с небольшим стартовым набором частых продуктов).
- Сопоставление названия с существующим `Ingredient` делает сам Claude: в промпт передаётся список названий из справочника вместе с фото, и модель сама решает, к какому существующему пункту отнести найденный продукт, либо помечает его как новый — отдельный fuzzy-matching алгоритм на бэкенде не нужен, это упрощает архитектуру. Итоговое решение всё равно подтверждает пользователь.
- Сравнение "есть/нужно" и агрегация списка покупок становятся точным сравнением по `ingredientId`, а не строк.

### Категории продуктов (ProductCategory) — контролируемый список

Чтобы группировка списка покупок по секциям (раздел 3, MVP-пункт 7) была предсказуемой — и для продуктов из справочника, и для ручных позиций — категория берётся из фиксированного перечисления, а не свободного текста:

- Молочные продукты
- Мясо и рыба
- Овощи и фрукты
- Крупы и бакалея
- Хлеб и выпечка
- Напитки
- Заморозка
- Бытовая химия
- Гигиена и красота
- Другое (catch-all, значение по умолчанию)

`Ingredient.category` заполняется этим списком при добавлении продукта в справочник (обязательно). Для ручных позиций (см. ниже) — то же самое перечисление, но в отдельном поле `ShoppingListItem.manualCategory`.

### Ручные позиции в списке покупок

`ShoppingListItem` дополнен полем `isManual`: пользователь может добавить в список покупок что угодно, не связанное с рецептами (зубная паста, бытовая химия) — такая позиция не привязана к `ingredientId` (поле nullable) и не участвует в сравнении с инвентарём, просто отображается в общем списке на неделю.

Чтобы было понятно, в какую секцию попадёт ручная позиция при группировке: форма добавления вручную включает **выбор категории** из того же списка `ProductCategory` (простой picker/select, не текстовое поле) — по умолчанию предзаполнено значение "Другое", если пользователь не выбрал ничего конкретного. Итоговая категория позиции при отображении списка: для обычных позиций — `Ingredient.category` через `ingredientId`, для ручных — `ShoppingListItem.manualCategory`.

### Ввод рецептов

MVP — только ручной ввод рецепта через форму: название, базовое число порций, ингредиенты из справочника, шаги приготовления (каждый шаг добавляется отдельно — текст + опциональное фото этого шага, а не одним большим полем текста, порядок — поле `order`, с возможностью переставить местами), и теги способа приготовления. Импорт по ссылке или фото рецепта с распознаванием — в V2.

### Единицы измерения

Единая система единиц на весь проект, разделённая по типу продукта (`unitType`):

| unitType | Единицы (`unit`) | Для чего |
|---|---|---|
| **WEIGHT** (вес) | г, кг | Мясо, овощи, крупы, сыпучие продукты — всё, что обычно взвешивают |
| **VOLUME** (объём) | мл, л | Жидкости: молоко, масло, вода, соки, бульон |
| **COUNT** (штучно) | шт | Яйца, овощи/фрукты поштучно (лук, помидор), упаковки как целая единица |

Правила:
- У каждого ингредиента/продукта единица подбирается по его природе: жидкости — только мл/л, всё остальное по умолчанию — г/кг, дискретные предметы — шт. `unitType` хранится отдельным полем, чтобы UI и валидация Zod могли ограничивать список `unit` соответствующим типу (нельзя выбрать "мл" для моркови).
- Хранится всегда в **базовой единице** (граммы для WEIGHT, миллилитры для VOLUME, штуки для COUNT), а "кг"/"л" — это только отображение в UI (например, показывать в кг, если значение ≥ 1000 г). Так агрегация в списке покупок — это просто суммирование чисел одного типа, без конвертации на лету.
- При распознавании фото холодильника Claude Vision должен определить не только продукт и количество, но и `unitType` (жидкость это или нет) — это часть промпта и Zod-схемы ответа.
- Примерная оценка количества (см. раздел 7, пункт 2) остаётся в силе для обоих типов — точность не нужна, важна только правильная единица измерения.

## 6. Архитектура и потоки

### Основные экраны

- **Сегодня** — главный экран/точка входа: слоты приёма пищи текущего дня (в порядке, заданном household в Профиле) с рецептом на каждый, время приготовления и бейджи способа готовки рядом с ним, кнопка "скушано" рядом с каждым — нажатие сразу отмечает приём пищи съеденным, а списание запасов предлагается отдельно и необязательно (см. раздел 6, потоки).
- **Меню на неделю** — конструктор/просмотр недели по дням, переход в любой день; у назначенного рецепта есть явное действие "убрать" (не только заменить), возвращающее слот в пустое состояние. В каждом дне одновременно видно максимум 3 карточки — считая и заполненные слоты (с рецептом), и пустые "+ добавить рецепт" (последние видны только Редактору/Организатору, у Участника пустые слоты не показываются вовсе, см. раздел 5 "Роли в household"). Если карточек для дня больше 3 — остальное доступно горизонтальной каруселью/свайпом внутри дня, а не сплошным списком вниз.
- **Рецепты** — **поле поиска по названию/ингредиенту** сверху, список карточек с бейджами способа приготовления, создание (пошагово, с фото на каждый шаг, базовым числом порций), **редактирование и удаление** (удаление — с подтверждением); на самой карточке рецепта — степпер порций, пересчитывающий количества ингредиентов. Карточка **"+ Добавить рецепт"** — первый элемент списка (перед всеми рецептами), а не кнопка/иконка в шапке экрана.
- **Инвентарь (домашние запасы)** — **поле поиска по названию продукта** сверху, список продуктов, ручное добавление, **редактирование количества/единицы и удаление** уже добавленных позиций, кнопка "фото холодильника" (позволяет добавить сразу несколько фото за один заход, не только одно).
- **Список покупок** — по умолчанию на текущую неделю, с фильтром по дням (Сегодня/Завтра/Вся неделя/Выбрать дни — см. поток ниже); отметка "куплено" мгновенная (без модалки), позиции можно редактировать прямо в списке; отдельная кнопка **"Добавить в запасы"** для массового переноса отмеченного в инвентарь; кнопка **"Поделиться"** — отправляет список текстом через Web Share API, с учётом текущего фильтра по дням; всё синхронизируется в реальном времени между всеми членами household; ручное добавление позиций.
- **Профиль / Настройки** — не отдельный таб, а экран, открывающийся по иконке в шапке любого из 5 основных экранов, включает и переключатель **светлой/тёмной темы** (см. подробности ниже).

Перед вёрсткой эти экраны стоит набросать в **Claude Design** (research preview, Pro/Max/Team/Enterprise) — описать текстом, доработать через чат и слайдеры, затем экспортировать результат прямо в Claude Code как отправную точку для реальной вёрстки на Next.js + Tailwind + shadcn/ui (см. этап 0 в разделе 8).

### Экран "Профиль / Настройки"

Содержимое:
- Имя и e-mail текущего пользователя (аватар — просто первая буква имени, без загрузки фото в MVP).
- Название household — редактируемое, необязательное; если не задано, показывается "Ваша семья".
- Список участников household (имя + e-mail каждого, и **роль** — Организатор/Редактор/Участник). Организатор видит рядом с каждым участником (кроме себя) выпадающий список для смены роли и кнопку "Удалить из household"; остальным ролям это не показывается, только сами роли как метки.
- Кнопка **"Пригласить"** → по нажатию раскрывает блок с самой ссылкой-приглашением, кнопкой "Скопировать" (и "Поделиться" через нативный Web Share API, если браузер поддерживает), и уже внутри этого раскрытого блока — вторичная кнопка **"Сгенерировать новую ссылку"** (инвалидирует предыдущую, перезаписывает `inviteCode` у household). До нажатия "Пригласить" кнопка перегенерации не видна вообще — незачем показывать второстепенное действие раньше, чем пользователь вообще увидел ссылку.
- **Слоты приёма пищи** (`MealSlot`) — редактируемый список: переименование, добавление нового слота, удаление, и **перетаскивание (drag-and-drop) для смены порядка** — этот порядок затем используется на экранах "Сегодня" и "Меню на неделю".
- **"Покинуть household"** — для участников, если их больше одного. Если пользователь единственный в household — кнопка недоступна (нет смысла выходить, только удаление аккаунта целиком, что не в MVP). Если выходит последний Организатор при наличии других участников — сначала нужно назначить Организатором кого-то ещё (см. раздел 5, "Роли в household").
- Кнопка "Выйти из аккаунта".
- **Переключатель темы** — Светлая / Тёмная / Как в системе. Личная настройка устройства (хранится локально, например `localStorage`), не часть данных household — у разных участников household тема может отличаться, и это ожидаемо.

Роли определяют, что видно/доступно: приглашение, удаление участников и смена ролей — только Организатору; управление слотами приёма пищи — Организатору и Редактору; настройки самого профиля (имя, выход из household/аккаунта) — всем (см. раздел 5, "Роли в household").

### Поток "создание household и приглашение участников"

1. Регистрация → автоматически создаётся `Household` без имени → пользователь становится единственным участником.
2. В Профиле — "Пригласить" → показывается ссылка вида `https://fridgemind.app/invite/{inviteCode}` → пользователь делится ей любым способом (мессенджер, копирование) вне приложения.
3. Человек, перешедший по ссылке:
   - если не зарегистрирован — проходит обычную регистрацию, после которой автоматически привязывается к `householdId` из ссылки (вместо создания своего нового household);
   - если уже зарегистрирован, но состоит в другом household — показывается предупреждение "Вы уже состоите в другой семье. Присоединение к этой заменит её" → подтверждение переносит пользователя в новый household (простая замена, без переноса/слияния данных — для MVP этого достаточно);
   - если уже состоит именно в этом household — ссылка просто открывает приложение как обычно.
4. Ссылка не имеет срока действия в MVP, но её можно инвалидировать через "Сгенерировать новую ссылку" в Профиле.

- **Server actions / API routes** для CRUD рецептов, инвентаря, меню, списка покупок.
- **Поток "фото холодильника"**: клиент делает или выбирает **одно или несколько фото** (кнопка "Добавить ещё фото" перед отправкой на распознавание, либо мультивыбор из галереи) → каждое сжимается на клиенте (browser-image-compression) → все фото отправляются **одним запросом** в API-роут → один вызов Claude API сразу со всеми изображениями и промптом на структурированный JSON-ответ (список продуктов + примерное количество), с инструкцией не дублировать один и тот же продукт, если он виден на нескольких фото → показ пользователю объединённого списка для подтверждения → запись в PantryItem. Один запрос с несколькими фото вместо нескольких последовательных — дешевле по токенам и позволяет модели сама разрешать пересечения между фото (например, один и тот же пакет молока в кадре и на общем плане, и крупным планом). Тем же способом (сжатие на клиенте перед загрузкой) обрабатываются и фото шагов рецепта — единая утилита компрессии для всех загружаемых изображений в приложении, чтобы не раздувать Supabase Storage.
- **Поток "список покупок"**: при формировании меню на неделю — берём RecipeIngredient всех запланированных `MenuDayMeal`, пересчитываем количество каждого под `servings` конкретного приёма пищи (см. раздел 5, "Порции"), для каждого такого вклада создаём запись `ShoppingListItemMeal` (количество + ссылка на конкретный `MenuDayMeal`, то есть на конкретный день), агрегируем по `ingredientId` с суммированием — это и есть `ShoppingListItem.quantity` на всю неделю. Вычитаем то, что уже есть в PantryItem (точное сравнение по `ingredientId`), остаток — то, что показывается как "нужно купить"; плюс пользователь может дописать туда произвольные позиции (`isManual`), не связанные со справочником и не привязанные к дням.
- **Поток "фильтр по дням в списке покупок"**: над списком — чипы "Сегодня" / "Завтра" / "Вся неделя" (по умолчанию) / "Выбрать дни" (открывает мультиселект из 7 дней недели). При выборе конкретных дней количество каждой позиции пересчитывается на лету: берём только те `ShoppingListItemMeal`, чей `MenuDayMeal` относится к выбранным дням, суммируем их `quantity`, вычитаем текущий остаток в PantryItem. Ручные позиции (`isManual`) не зависят от фильтра и показываются всегда, независимо от выбранных дней — они не привязаны к конкретному дню. Отметка "куплено" (`isBought`) не зависит от фильтра — это состояние всей позиции целиком, а не отдельно на каждый день.
- **Поток "убрать рецепт из дня меню"**: у назначенного в слот рецепта есть действие "убрать" (например, через долгое нажатие/свайп или маленькую иконку на карточке слота) — по подтверждению соответствующая запись `MenuDayMeal` удаляется, слот в UI возвращается в пустое состояние "+ добавить рецепт". Если этот приём пищи уже был отмечен "скушано" (`isEaten = true`) — удаление всё равно разрешено (снимает план на будущее/меняет историю дня), но ранее подтверждённое списание из инвентаря (если оно было) автоматически не отменяется — та же логика "ничего не делаем незаметно за пользователя", что и в остальном приложении.
- **Офлайн** (реализация — этап 9, пересмотрено): офлайн — **только чтение**, синхронизации отложенных изменений (outbox-очереди мутаций) в MVP нет — сознательное сокращение, см. раздел 7, пункт 5. Три слоя:
  - **Serwist** кэширует статику/страницы (runtime-кэш `defaultCache` из `@serwist/next/worker`), а на любую неудавшуюся навигацию без кэша отдаёт precache'нутую страницу `/~offline` (`fallbacks` в `app/sw.ts` + `additionalPrecacheEntries` в `next.config.mjs`; маршрут добавлен в `PUBLIC_PATHS` middleware — SW precache'ит его без сессии).
  - **Dexie** (`lib/offline-db.ts`) хранит типизированные view-снапшоты экранов: `menuWeeks` (id — понедельник недели), `menuDays` (id — дата), `shoppingLists` (id — понедельник недели), `recipeLists` (id — `"all"`), `recipes` (id — recipeId), `pantry` (id — `"all"`). Запись — клиентский `<OfflineSnapshot />` на страницах Сегодня/Меню/День/Рецепты/Рецепт/Запасы/Список покупок: кладёт в Dexie те же данные, что уже отрисовал серверный компонент (без дублирующего запроса), перезаписывая снапшот при каждом успешном онлайн-визите и каждом обновлении RSC-данных (revalidatePath, Realtime). Кэшируется то, что реально открывали: список рецептов — целиком (карточки), детали рецепта — по мере посещения.
  - **`/~offline`** (`components/offline/OfflineApp.tsx`) — полностью клиентская страница: по `window.location.pathname` (URL остаётся исходным, SW лишь подменяет ответ) выбирает, какой снапшот показать, и рендерит упрощённый read-only вариант экрана с временем сохранения (`cachedAt`). Навигация внутри неё — обычные `<a>`, не `next/link`: каждый переход — полная загрузка документа через SW.
  - **Индикатор**: `useOnline` (`lib/hooks/use-online.ts`) + фиксированный баннер `OfflineBanner` в layout и на `/~offline` — пользователь видит, что данные сохранённые. Отметка "куплено" без сети сразу показывает ошибку, не притворяясь сохранённой.
  - Инвалидация: TTL нет — "последний онлайн-визит побеждает", снапшоты старше 30 дней вычищаются при записи.
  - **Сокращения этапа 9 (доделать при необходимости)**: фильтр по дням в офлайн-списке покупок недоступен (только вся неделя); офлайн-просмотр рецепта — без степпера порций и фото шагов; мутации офлайн не буферизуются: любой вызов server action без сети показывает общую ошибку сети в форме (callAction/guardFormAction из lib/form-state.ts — оборачивают все императивные вызовы и useActionState-экшены), плюс страховочный app/error.tsx вместо голого "Application error"; кэш (Dexie-снапшоты + пользовательские runtime-кэши SW: pages/RSC/next-data/apis/next-image — SW кэширует HTML по URL без учёта сессии) очищается при явном "Выйти из аккаунта" и при смене household (скоуп по householdId), но **не** при простом закрытии приложения; при этом сам `/~offline` показывает снапшоты только при наличии auth-cookie Supabase (проверка присутствия cookie, а не `getSession()` — офлайн истёкший access token не обновить, и getSession() отказал бы легитимному пользователю; выход из аккаунта удаляет cookie и чистит кэш), так что человек без входа на общем устройстве чужие данные не увидит.
- **Состояние загрузки при переходе между экранами (skeleton screens)**: вместо спиннера или пустого экрана — серые skeleton-плейсхолдеры, повторяющие форму реального контента экрана (карточка рецепта/приёма пищи, строка инвентаря, позиция списка покупок, блок профиля), чтобы не было "прыжка" layout, когда данные подгрузятся. Технически — файл `loading.tsx` на каждый маршрут App Router (Next.js оборачивает страницу в Suspense автоматически и показывает `loading.tsx`, пока серверный компонент ждёт данные). Так как большая часть экранов кэшируется в Dexie (см. выше, "Офлайн") — skeleton реально виден только при отсутствии кэша (первая загрузка) или при заметной задержке сети, а не при каждом переходе между табами, если данные уже есть локально.
- **Поток "отметил, что скушал"**: нажатие "скушано" сразу помечает `MenuDayMeal.isEaten = true` (`eatenAt`, `eatenByUserId`) — без каких-либо подтверждений, это мгновенное действие. Отдельно, поверх этого, система показывает необязательный экран-предложение "Списать использованные продукты?": берёт `RecipeIngredient` этого рецепта, пересчитывает количества под `MenuDayMeal.servings` (см. раздел 5, "Порции") и показывает только эти количества — **без сравнения с текущими остатками** ("есть дома"/"нужно купить" здесь не показываются, это отдельная концепция для экрана "Просмотр дня", не для этого потока). На этом же экране есть **степпер числа порций** (по умолчанию — текущее `MenuDayMeal.servings`) — если по факту готовили не столько, сколько планировали, пользователь меняет значение, и список ингредиентов сразу пересчитывается под новое количество порций (та же пропорция, что и в разделе 5). У пользователя два равнозначных варианта: **"Списать"** (можно ещё и вручную поправить итоговые цифры по каждому ингредиенту перед подтверждением — на случай, если пропорция "не как в рецепте", что-то доели не всё и т.п.; при подтверждении `PantryItem.quantity` уменьшается, не ниже нуля, и заодно обновляется `MenuDayMeal.servings` на выбранное здесь значение — чтобы в истории осталось реальное, а не изначально запланированное число порций) или **"Пропустить"** — закрыть экран без каких-либо изменений в инвентаре (в этом случае `MenuDayMeal.servings` тоже не меняется). Отметка "скушано" никак не зависит от выбора здесь — она уже применена в момент нажатия кнопки.
- **Поток "отметил, что купил"**: пользователь ставит галочку `isBought` у позиции в `ShoppingListItem` (записывается `boughtByUserId`) — это **мгновенное действие, без какого-либо подтверждения или всплывающего окна**: галочка/зачёркивание сразу, и всё. Это специально сделано лёгким, потому что человек в магазине отмечает так одну позицию за другой подряд, и модалка на каждую была бы утомительна. Инвентарь на этом шаге не трогается.
- **Поток "редактирование позиции списка покупок"**: у любой позиции (и обычной, привязанной к `ingredientId`, и ручной) можно открыть редактирование — поменять название (только у ручных), количество, единицу (тоже только у ручных — у обычных единица задана справочником и должна совпадать с базовой единицей для сравнения с запасами). Изменение количества здесь не пересчитывает автоматически весь недельный план — это ручная корректировка конкретно списка покупок (например, "на самом деле нужно на 2 упаковки больше"). Реализация (этап 8): правка попадает в `ShoppingListItem.quantity` и **переживает пересинхронизацию списка с меню** — `syncWeekItems` сравнивает не количество, а вклады (`ShoppingListItemMeal`), и переписывает quantity только при реальном изменении плана; недельный вид считает "нужно купить" от `quantity`, а фильтры по дням распределяют его по дням пропорционально вкладам — чтобы день не мог требовать больше, чем вся неделя. Рост количества при правке снимает `isBought`/`addedToPantry` (то же правило, что при росте потребности в `syncWeekItems`) — иначе излишек молча терялся бы; снятие галочки "куплено" флаг `addedToPantry` не трогает — уже перенесённое в запасы не должно переноситься повторно. **Удаление позиции — только для ручных**: позицию из меню синхронизация пересоздала бы при следующем чтении, поэтому у них удаления нет (из плана они уходят правкой меню).
- **Поток "массовое обновление инвентаря из списка покупок"**: отдельная кнопка **"Добавить в запасы"** (видна, когда есть хотя бы одна позиция с `isBought = true` и `addedToPantry = false`) → открывает экран со списком всех таких позиций и их количеством → пользователь может поправить цифры по каждой (купил больше/меньше, чем планировал) → по подтверждению каждая позиция создаёт или пополняет соответствующий `PantryItem` и помечается `addedToPantry = true`. Ручные позиции (`isManual`, без `ingredientId`) в этом экране не участвуют — им нечего добавлять в инвентарь. Можно и пропустить конкретную позицию в этом экране (например, снять галочку) — тогда она останется `addedToPantry = false` и снова появится здесь в следующий раз.
- **Поток "общий список покупок в реальном времени"**: клиент подписывается на изменения `ShoppingListItem` своего `householdId` через Supabase Realtime (вебсокет). Когда один участник добавляет позицию, отмечает "куплено" или правит количество — остальные участники household видят изменение почти мгновенно, без обновления страницы. Полученные через Realtime изменения применяются и к офлайн-кэшу в Dexie, чтобы при потере сети последнее известное состояние списка оставалось актуальным. Конфликтов при одновременном редактировании одной и той же позиции в MVP не решаем отдельно — Realtime обновления идут по принципу "последняя запись побеждает" (last write wins), для семейного использования этого достаточно.

## 7. Принятые решения

1. **Шаринг между пользователями** — пересмотрено: нужен уже в MVP. Всё приложение общее для household (семья, без ограничения по числу участников — может быть 2, может больше) — не только список покупок, а запасы, меню и рецепты тоже. Список покупок дополнительно синхронизируется в реальном времени (Supabase Realtime), чтобы один человек мог дописать позицию, пока другой уже в магазине и отмечает купленное. **Роли внутри household — тоже уже в MVP** (пересмотрено ещё раз): Организатор/Редактор/Участник, подробности в разделе 5 ("Роли в household"). Более гибкие поштучные права на каждое действие — в V2, если понадобится.
2. **Точность граммовки** — примерная оценка (не точный вес), и в рецептах, и при распознавании фото холодильника.
3. **Приёмы пищи** — пересмотрено: не фиксированный список (завтрак/обед/ужин и т.д.), а полностью пользовательские слоты (`MealSlot`) — household сам называет, добавляет, удаляет и сортирует их drag-and-drop'ом в Профиле. Стартовый набор при создании household — "Завтрак/Обед/Ужин", дальше свободно редактируется. Набор общий на весь household, а не индивидуальный для каждого участника.
4. **Монетизация** — сейчас не нужна: приложение личное, для себя. Рекомендация: не строить биллинг заранее, но заложить простой счётчик вызовов Claude Vision API (например, лимит N фото-распознаваний в день) — сами вызовы стоят денег даже для одного пользователя, и без лимита можно случайно накрутить счёт при частом фотографировании холодильника. Если позже решишь открыть доступ другим household — модель данных (householdId везде) уже готова к добавлению подписки/тарифов без переписывания архитектуры.
5. **Офлайн — только чтение (этап 9)**: формулировка MVP (раздел 3, пункт 11) требует только просмотра сохранённого меню/рецептов/списка покупок, поэтому буферизация офлайн-мутаций с последующей синхронизацией сознательно **не делается** — она требует outbox-очереди и конфликтует с принятым в этапе 8 last-write-wins через Supabase Realtime. Без сети действия недоступны (отметка "куплено" сразу сообщает об этом, остальные мутации падают с ошибкой сети), просмотр работает из Dexie-снапшотов через `/~offline` (см. раздел 6, "Офлайн"). Если офлайн-запись когда-нибудь понадобится — начинать с отметки "куплено" как самого частого офлайн-сценария (магазин без связи).

| 9 | PWA: manifest, иконки, Serwist, офлайн-кэш, установка на телефон | 2–3 дня |
| 10 | Полировка UI, тестирование на мобильном (в т.ч. с нескольких аккаунтов household одновременно — реальный сценарий "один дома, другой в магазине"), багфиксы, деплой | 3–5 дней |

**Итого MVP (включая этап 0 с дизайном): примерно 6–8 недель в режиме полного дня, или 10–13 недель в режиме вечера/выходные (10–15 часов в неделю).**

С Claude Code скорость написания кода заметно выше, чем без ИИ, но время всё равно уйдёт на: тестирование на реальном телефоне (PWA-специфика — установка, офлайн-режим, работа камеры), подбор промпта для стабильного распознавания продуктов на фото, и итерации по UX меню/списка покупок — это обычно занимает больше времени, чем сам код.

## 10. Технические соглашения разработки

### Naming: camelCase на фронте, snake_case в Postgres
В Prisma-схеме поля объявляются в camelCase (`createdAt`, `expiresAt`), а маппинг на реальные колонки/таблицы в Postgres — через `@map("created_at")` на уровне поля и `@@map("recipe_ingredients")` на уровне модели. Prisma Client отдаёт в код уже camelCase-объекты, так что во фронтенде и server actions camelCase используется everywhere, а в самой БД — snake_case. Это надо зафиксировать в схеме с самого первого коммита, чтобы не переименовывать колонки позже.

### Типизация
- TypeScript в `strict` режиме с первого дня (`strict: true` в tsconfig), без `any` кроме явно обоснованных мест.
- Типы для слоя БД — автоматически из Prisma Client, отдельно руками не дублируются.
- Для всего, что приходит **извне и не гарантированно валидно** (JSON-ответ от Claude Vision, тело запросов в API-роутах, данные форм) — runtime-валидация через **Zod**, а TS-типы для этих объектов выводятся из Zod-схем (`z.infer<...>`), а не пишутся отдельно. Это особенно важно для ответа ИИ на фото холодильника: модель может вернуть неожиданный формат, и без валидации это уйдёт прямо в БД. В частности, `unitType` и `unit` из раздела 5 — это `z.enum(...)`, а не свободная строка, чтобы Claude Vision не мог вернуть, например, "мл" для моркови.
- Общие типы (Recipe, PantryItem, MenuDayMeal и т.д.) держать в одном месте (`/lib/types` или рядом с Zod-схемами) и переиспользовать между сервером и клиентом — благо Next.js App Router это позволяет в рамках одного проекта без отдельного пакета.
- Skeleton-компоненты (см. раздел 6, "Состояние загрузки") держать в `components/skeletons/` — один компонент на форму карточки/строки (рецепт, слот приёма пищи, позиция инвентаря/списка покупок, блок профиля), переиспользуются и в `loading.tsx` каждого маршрута, и как инлайн-заглушка при дозагрузке (например, следующая страница списка), а не дублируются в каждом месте отдельно. Базовый прямоугольник — примитив `Skeleton` из `components/ui`. `loading.tsx` лежит **рядом со своей страницей внутри route group** (`app/(app)/inventory/loading.tsx`, не `app/inventory/`) — иначе Next его никогда не покажет.

### UI-компоненты
- Примитивы (кнопки, инпуты, лейблы, карточки и т.д.) — только из `components/ui` (shadcn/ui: `cva`-варианты + `cn()`; добавление новых — через `npx shadcn add <name>`). Экраны и составные компоненты **не пишут длинные tailwind-цепочки для примитивов** — если строка классов повторилась второй раз, она выносится в примитив или его вариант.
- Составные/доменные компоненты — в `components/<домен>/` (`nav/`, `auth/`, `recipes/`, ...). Server actions — в `lib/actions/<домен>.ts` (`auth.ts`, `invite.ts`, ...), не колокацией у маршрутов — экшены переиспользуются между экранами. В `app/` живут только страницы, layouts и роуты.
- **Чтения (read-запросы) не выносятся в `lib/queries/` по инерции — только когда есть повод.** Прагматично, не ради симметрии: голый `prisma.*.findUnique/findMany` с единственным вызывающим и без преобразования данных остаётся инлайном в серверном компоненте. Выносим в `lib/queries/<домен>.ts` (обычный async-модуль, **не** `"use server"` — это чтение, а не мутация), когда выполняется хотя бы одно: (1) есть нетривиальный шейпинг/бизнес-логика поверх запроса; (2) запрос переиспользуется на ≥2 экранах; (3) в логике есть что покрыть юнит-тестом. Чистые расчёты внутри такого запроса — отдельной тестируемой функцией в `lib/<домен>.ts` (как `matchCounts` в `lib/recipes.ts`). View-флаги (`hasRole(...)`, `canLeaveHousehold(...)`) считаются в самой странице, а не в query.
- `"use client"` — минимально: только там, где нужен стейт/эффекты/браузерные API. Форма — клиентский компонент, страница-обёртка — серверная.
- Формы — **всегда** через server actions + `useActionState` (React 19), не через контролируемые `useState`-поля и ручные `loading`/`error`. Состояние любой формы — единый дженерик `FormState<TValues, TData>` из `lib/form-state.ts` (`error: string | null` — общая ошибка, `fieldErrors?` — ошибки по полям, `values?` — восстановление введённого после ошибки через `defaultValue={state.values?...}`, `data?` — полезная нагрузка успеха без redirect'а). Начальное состояние — общий `initialFormState`. **Ошибки валидации показываются под своим полем** (`fieldIssues()` раскладывает Zod-issues по верхнеуровневым полям схемы; сообщения в схемах — по-русски): у `Input` для этого есть проп `error` (красная рамка + текст под полем), для не-Input контролов — локальный `FieldError`. Общие ошибки (права, сбой) — заметным блоком в начале формы; `firstIssue()` — для простых форм с одним сообщением. Экшен: FormData → Zod-схема → работа → `redirect()` либо `FormState`. Контролируемые поля — только когда значение реально нужно на клиенте до сабмита (поиск, живой пересчёт).
- У полей ввода (Input/Textarea и их переопределения className) размер шрифта на мобильном — не меньше 16px (`text-base`): iOS Safari автоматически зумит страницу при фокусе на поле с меньшим шрифтом, и экран «прыгает».
- Цвета/радиусы — только токены темы (`bg-primary`, `text-destructive`, ...), без сырых значений в классах. Токены объявлены в `app/globals.css` + `tailwind.config.ts` (включая стандартные shadcn: `input`, `ring`, `secondary`, `destructive`, `popover`).
- Пропсы компонентов — явный `type Props = { ... }` рядом с компонентом; `React.FC` не используется. Данные с сервера типизируются типами из `lib/types` (Prisma/view-типы), а не инлайн-объектами.
- Пропсы страниц/layout'ов — через сгенерированные Next'ом типизированные роуты: `type Props = PageProps<"/invite/[code]">` (и `LayoutProps<...>` для layout'ов), а не рукописные `{ params: Promise<...> }` — параметры остаются в синхроне с реальным путём автоматически.
- Импорты из `react` — только именованные, без `import * as React`: значения обычным импортом, типы — `import type { ReactNode }` (или inline `type X` в смешанном импорте). `React.Xxx`-префиксы в коде не используются. Файлы, сгенерированные `npx shadcn add`, приводятся к этому стилю сразу после генерации (вместе с адаптацией цветов к теме).
- `type` по умолчанию, `interface` — только там, где нужен declaration merging (например, расширение глобального `WorkerGlobalScope` в `app/sw.ts`) — закреплено ESLint-правилом `consistent-type-definitions`.

### Комментарии
Не добавлять многословные пояснительные/обосновывающие комментарии («что и почему делает код», «вместо X», «из макета»). Код должен читаться сам; комментарий оправдан только там, где поведение неочевидно и без него легко ошибиться (нетривиальный инвариант, обход бага, ссылка на решение). Дефолт — без комментария.

### ESLint
Flat config (`eslint.config.mjs`): ESLint 9 + `next/core-web-vitals` + `next/typescript` (через FlatCompat). Команды: `npm run lint` / `npm run lint:fix` (`next lint` не используется — deprecated). Поверх пресетов:
- `simple-import-sort/imports|exports` — порядок импортов чинится автофиксом, руками не сортируем;
- `@typescript-eslint/consistent-type-imports` (inline `type`-модификаторы), `consistent-type-definitions: type`;
- запрет дефолтного импорта `react` (`no-restricted-imports`).
Отступления от правил — только точечным `eslint-disable-next-line` с пояснением причины.

Перед каждым коммитом линтер запускается автоматически — husky pre-commit хук (`.husky/pre-commit` → `npm run lint`); коммит с ошибками линта не проходит.

### Тесты
Полное E2E-покрытие для personal-use проекта избыточно, поэтому фокус на местах, где реально возникают баги:
- **Vitest** — юнит-тесты на бизнес-логику: агрегация ингредиентов по неделе, пересчёт количества ингредиентов при смене числа порций (включая округление для COUNT), сравнение "есть дома / нужно купить", конвертация единиц измерения, парсинг и валидация JSON-ответа Claude Vision (в том числе на "плохих" ответах — модель что-то не распознала или вернула лишнее поле).
- **Playwright** — небольшой набор smoke-тестов на критичные сквозные сценарии: добавить рецепт → собрать меню на неделю → получить список покупок; распознать фото → подтвердить продукты → увидеть их в инвентаре; установка PWA и чтение сохранённого меню офлайн.
- Тесты пишутся по мере готовности фич (не отдельным этапом в конце) — проще всего добавлять их сразу за той частью из раздела 8, к которой они относятся.
